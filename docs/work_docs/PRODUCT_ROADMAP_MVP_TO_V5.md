# 产品路线图：MVP 到 V5

## 1. 文档目的

本文件用于记录本项目从最小可运行版本到后续演进版本的阶段计划。

当前口径：

- 本轮先不继续 coding，只写 doc
- 先把范围、边界、任务、版本顺序说清楚
- 等容量刷新后，再按本文件推进实现

---

## 2. 总体原则

### 2.1 项目定位

这是一个美股选股提醒 Bot，不是自动交易系统。

核心闭环是：

`股票池 -> feed -> 实时扫描 -> 规则筛选 -> 排名输出 -> Telegram / Discord 提醒`

### 2.2 当前策略原则

当前阶段不做研究型量化，不做复杂 alpha，不做回测。

当前只做最简单、最能落地的实时提醒框架。

### 2.3 版本演进原则

- `MVP` 先跑通
- `V1` 先稳定
- `V2` 再增强使用体验
- `V3` 再增强策略表达能力
- `V4` 再增强系统可靠性
- `V5` 再考虑研究化和平台化

---

## 3. MVP

### 3.1 目标

证明系统闭环可运行。

### 3.2 范围

- 使用 Finnhub 作为唯一 feed
- 扫描 P0 / P1 股票池
- 按 `day_change_pct` 排序
- 输出 Top 10
- 能发 Telegram
- 能发 Discord
- 支持每小时自动跑一次
- 支持手动 `/scan`

### 3.3 不做

- 回测
- beta
- 多因子
- K 线
- 自动交易
- Web 页面
- 数据库

### 3.4 MVP 交付物

- PRD
- 股票池文件
- 基础配置
- Bot 项目骨架
- 基础扫描引擎
- 基础消息推送

### 3.5 MVP 验收

- 本地可运行
- 可以真实拉到 Finnhub 数据
- Telegram / Discord 至少一个通道成功出消息
- 扫描结果可读

---

## 4. V1

### 4.1 目标

把 MVP 从“能跑”提升到“可持续使用”。

### 4.2 重点

- 稳定每小时自动扫描
- 补齐 `/status`、`/help`
- 明确状态机
- 处理 API 限流与部分失败
- 输出内容统一为中文
- Linux / AWS 常驻部署跑通

### 4.3 V1 交付物

- 稳定版 PRD
- 中文消息模板
- systemd 部署方案
- 状态持久化文件
- 错误与日志规范
- 最小测试集

### 4.4 V1 验收

- AWS 上可常驻
- 每小时扫描稳定
- `/scan`、`/status`、`/help` 可用
- 输出中文
- 限流或单标的失败不会导致整体崩溃

---

## 5. V2

### 5.1 目标

把 V1 从“可用”提升到“更像产品”。

### 5.2 重点

- 引入更明确的过滤规则配置
- 增加提醒去重与冷却机制
- 增加盘前 / 盘中 / 盘后状态区分
- 增加更清晰的排名说明
- 增加股票池维护流程

### 5.3 可选能力

- Top N 可配置
- P0 / P1 权重更明确
- 通知模板可配置
- 不同市场状态下不同推送策略

### 5.4 V2 验收

- 用户能理解为什么某股票被选中
- 重复提醒显著减少
- 配置改动不需要重写核心代码

---

## 6. V3

### 6.1 目标

把系统从单一裸跑排序，提升为可表达简单策略组合的扫描平台。

### 6.2 重点

- 引入策略配置层
- 支持多种过滤器组合
- 支持多种排序器组合
- 引入更明确的评分解释
- 保持不做回测，但允许更丰富的实时规则

### 6.3 可选能力

- 行业维度过滤
- ETF / 个股分池处理
- 风险标签
- 简易事件标签

### 6.4 V3 验收

- 新增一个策略不需要大改框架
- 输出结果能解释“因为什么规则入选”

---

## 7. V4

### 7.1 目标

把系统从“单机可用”提升为“长期稳定运维”。

### 7.2 重点

- 更强的日志与告警
- 更完整的运行监控
- 更稳健的异常恢复
- 更明确的配置隔离
- 更完整的部署脚本

### 7.3 可选能力

- 失败重试策略升级
- 健康检查
- Bot 通道失效告警
- 扫描结果落地归档

### 7.4 V4 验收

- 长时间运行不容易失控
- 出错时能快速定位原因
- 部署与重启流程清晰

---

## 8. V5

### 8.1 目标

把项目从提醒工具，推进到研究与策略协同平台。

### 8.2 重点

- 接入更多 feed
- 引入历史数据研究链路
- 拆分研究文档与生产配置
- 为未来回测或策略研究预留接口

### 8.3 注意事项

V5 才适合认真考虑：

- 回测
- 多因子
- 更复杂状态机
- 更复杂风控表达

这些内容不应该提前压进 MVP / V1。

---

`文档冻结 + 版本规划阶段`
