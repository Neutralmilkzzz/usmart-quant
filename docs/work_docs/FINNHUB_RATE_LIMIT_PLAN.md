# Finnhub 免费额度限流方案

## 1. 背景

当前 Finnhub 免费额度为：

`60 API calls / minute`

当前程序在一次扫描中会对每只股票请求多个接口，因此在 `P0 / P1` 全池扫描时，极易触发 `HTTP 429 Too Many Requests`。

实测现象：

- 当前启用股票池为 `76` 只
- 单次扫描可完成，但通常以 `PARTIAL_SUCCESS` 结束
- `profile` 和 `metric` 接口频繁触发 `429`

---

## 2. 当前判断

这不是用户操作错误，也不是 Telegram 或 systemd 引起的问题。

核心原因是：

- 当前请求模型属于突发式扫描
- 免费额度属于按分钟限速
- 现有扫描没有显式限流器

因此，`429` 属于当前实现下的预期结果。

---

## 3. 项目决策

老板当前决策如下：

1. 本项目属于低频场景
2. 不要求在 1 分钟内扫完整个股票池
3. 可以接受扫描过程拉长
4. 不优先通过缩小股票池来规避限流
5. 优先采用“慢慢扫完”的方式，适配 Finnhub 免费额度

因此，V1 / V2 阶段的正式口径是：

`接受低频长扫描，以 55 到 60 次 API 调用 / 分钟为运行约束。`

---

## 4. 目标口径

后续扫描设计应满足：

- 扫描过程中尽量不触发 `429`
- 即使股票池扩大到 `229` 只或 `420` 只，仍可按节流方式逐步完成
- 扫描总时长可以增长，但系统应保持稳定和可预测

这意味着项目优先优化的是：

- 速率控制
- 请求调度
- 扫描时长可接受性

而不是优先优化：

- 单次扫描完成速度
- 高频触发
- 复杂实时因子

---

## 5. 实施原则

### 5.1 限速目标

建议以保守值运行：

`55 calls / minute`

而不是打满理论上限 `60 calls / minute`。

理由：

- 留出 Telegram / 其他偶发请求的余量
- 留出重试和时钟误差的余量
- 降低连续分钟边界造成的误伤

### 5.2 扫描方式

扫描应改为：

- 按队列逐只处理
- 按速率控制发请求
- 股票池大时允许扫描时间延长

不要求：

- 1 分钟内必须扫完整池

### 5.3 接口优先级

后续优化时应区分：

- `quote`：高频实时接口
- `profile`：低频静态接口
- `metric`：低频或中频接口

即使最终继续保留三类接口，也应按不同频率和不同调度策略处理。

---

## 6. 粗略容量测算

假设每只股票仍需多次请求，则扫描容量应按分钟预算来排队，而不是一次性全部发出。

举例：

- 若预算为 `55 calls/min`
- 若单只股票平均消耗 `3` 次请求

则每分钟大约只能稳定处理：

`18` 只股票

粗略估算：

- `76` 只股票约需 `5` 分钟
- `229` 只股票约需 `13` 分钟
- `420` 只股票约需 `24` 分钟

以上仅为预算级估算，实际时长还会受：

- 重试
- 网络波动
- 某些接口失败

影响。

但从产品角度看，这个时长在“低频提醒系统”场景下是可接受的。

---

## 9. 当前结论

当前项目对 Finnhub 免费额度的处理原则，不是“减少 ambition”，而是“接受慢速完成”。

后续改造方向应以这条原则为准：

`宁可扫描更慢，也尽量不要继续用突发式请求把免费额度打爆。`
